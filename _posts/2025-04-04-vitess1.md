vitess는 mysql을 기반으로 구축된 클라우드 네이티브 수평 확장형 분산 데이터베이스 시스템입니다. 이번 “vitess 분석” 시리즈는 vitess에 대한 동작 원리를 전반적으로 탐색해봅니다. 가장 먼저 확인할 부분은 서버를 열고 vitess에서 커넥션 연결을 대기하는 부분입니다. 이 부분은 vitess 구성 요소 중 vtgate라는 요소가 담당하며 vtgate는 mysql 쿼리를 클라이언트로 받는 프록시 서버 역할을 합니다. 따라서 vtgate를 먼저 분석해보도록 하겠습니다.

# vtgate pod의 vitess 진입점 분석

vtgate는 vitess 내에서 트래픽을 vttablet 서버로 라우팅하고 통합된 결과를 클라이언트에 다시 반환하는 프록시 서버입니다. mysql 프로토콜을 지원하여 마치 mysql 서버인 것처럼 연결할 수 있습니다. kubernetes에 vitess operator를 배포해보면 vtgate pod가 호스팅 된 것을 확인할 수 있습니다. 아래는 vtgate에 대한 deployment를 출력한 결과물입니다.

# vitess 코드 내부: mysql 프록시 서버 실행

지금까지 kubernetes 상에 vitess를 띄우고 어떻게 바이너리를 실행하여 코드 내부 상의 위치를 탐색했습니다. 이번에는 코드 내에서 실제 로직이 호출되는 위치를 파악해보겠습니다.

Main command가 실행되면 다음이 호출됩니다. Main command의 로직은 토폴로지 서버를 생성하고 vtgate의 초기화를 진행합니다 

```go
// https://github.com/vitessio/vitess/blob/94fdc736eae8928a8fdde44e9ec9c3bee1868d6f/go/cmd/vtgate/cli/cli.go#L141
func run(cmd *cobra.Command, args []string) error {
	defer exit.Recover()

	// ✅ 환경변수 초기화
	servenv.Init()

	// ✅ topology server 초기화(etcd)
	// Ensure we open the topo before we start the context, so that the
	// defer that closes the topo runs after cancelling the context.
	// This ensures that we've properly closed things like the watchers
	// at that point.
	ts := topo.Open()
	defer ts.Close()

	// topo.Server를 기반으로한 resilientServer 생성
	resilientServer = srvtopo.NewResilientServer(ctx, ts, srvTopoCounts)

	// ...

	// 환경 변수 설정
	env, err := vtenv.New(vtenv.Options{
		// ...
	})
	// ...

	// ✅ vtgate 초기화
	// pass nil for HealthCheck and it will be created
	vtg := vtgate.Init(ctx, env, nil, resilientServer, cell, tabletTypes, plannerVersion)
	
	// ...
}
```

이 코드는 환경 변수 초기화, topology 서버(ex: etcd) 초기화, vtgate 초기화를 담당합니다. 이 함수에서 가장 중요한 부분은 `vtgate.Init`입니다. 해당 부분에서 vtgate 패키지의 초기화를 진행합니다. 함수를 다 읽어볼 필요 없이 초기 실행에서는 ✅ 만 확인하면 됩니다.

```go
// https://github.com/vitessio/vitess/blob/94fdc736eae8928a8fdde44e9ec9c3bee1868d6f/go/vt/vtgate/vtgate.go#L251
func Init(
	ctx context.Context,
	env *vtenv.Environment,
	hc discovery.HealthCheck,
	serv srvtopo.Server,
	cell string,
	tabletTypesToWait []topodatapb.TabletType,
	pv plancontext.PlannerVersion,
) *VTGate {

	// TabletGateway 생성
	// vttablet 연결 관리 및 헬스 체크
	gw := NewTabletGateway(ctx, hc, serv, cell)
	gw.RegisterStats()
	if err := gw.WaitForTablets(ctx, tabletTypesToWait); err != nil {
		log.Fatalf("tabletGateway.WaitForTablets failed: %v", err)
	}

	// txConn 생성
	// 트랜잭션 상태 관리
	tc := NewTxConn(gw, getTxMode())
	// 트랜잭션을 지원하는 샤딩 쿼리 실행기(?) 아직 뭔지 모름
	// ScatterConn depends on TxConn to perform forced rollbacks.
	sc := NewScatterConn("VttabletCall", tc, gw)
	// 분산 트랜잭션
	// TxResolver depends on TxConn to complete distributed transaction.
	tr := txresolver.NewTxResolver(gw.hc.Subscribe(), tc)
	srvResolver := srvtopo.NewResolver(serv, gw, cell)
	// 키스페이스/타입/샤드 분해 및 라우팅
	resolver := NewResolver(srvResolver, serv, cell, sc) 
	// binlog 스트리밍
	vsm := newVStreamManager(srvResolver, serv, cell)
	
	// 토폴로지 서버 초기화
	ts, err := serv.GetTopoServer()
	
	// 스키마 트래커 생성
	var st *vtschema.Tracker
	if enableSchemaChangeSignal {
		st = vtschema.NewTracker(gw.hc.Subscribe(), enableViews, enableUdfs, env.Parser())
		addKeyspacesToTracker(ctx, srvResolver, st, gw)
		si = st
	}

	// executor 생성: 쿼리 실행 및 분석
	executor := NewExecutor(
		// ...
	)
	
	// ✅ vtgate 생성
	vtgateInst := newVTGate(executor, resolver, vsm, tc, gw)
	
	// ✅ 서버 실행
	servenv.OnRun(func() {
		// grpc/mysql 서버 등에 서비스 등록
		for _, f := range RegisterVTGates {
			f(vtgateInst)
		}
		// 스키마 트래커 시작
		if st != nil && enableSchemaChangeSignal {
			st.Start()
		}
		// 트랜잭션 리졸버 시작
		tr.Start()
		// ✅ mysql 서버 시작
		srv := initMySQLProtocol(vtgateInst)
		if srv != nil {
			servenv.OnTermSync(srv.shutdownMysqlProtocolAndDrain)
			servenv.OnClose(srv.rollbackAtShutdown)
		}
	})
	
	// ...
	
	return vtgateInst
}
```

✅만 확인해보면 해당 함수는 단순하게 서버 생성 및 실행 로직을 담고 있습니다. 간단하게는 다음과 같이 요약할 수 있습니다.

- vtgate를 생성
- 서버를 `initMySQLProtocol`을 호출하여 실행

그럼 서버 실행하는 부분인 `initMySQLProtocol`에서 어떤 작업이 일어나는지 살펴보겠습니다. 이 함수는 vtgate를 mysql 서버처럼 보이게 만드는 TCP/socket 리스너 초기화 함수입니다.

```go
// https://github.com/vitessio/vitess/blob/94fdc736eae8928a8fdde44e9ec9c3bee1868d6f/go/vt/vtgate/plugin_mysql_server.go#L492
// initMySQLProtocol starts the mysql protocol.
// It should be called only once in a process.
func initMySQLProtocol(vtgate *VTGate) *mysqlServer {
	// ...

	// ✅ TCP 리스너 생성
	// Create a Listener.
	var err error
	srv := &mysqlServer{}
	srv.vtgateHandle = newVtgateHandler(vtgate) // ✅ 연결 핸들러 생성: 커넥션 이벤트 발생 시 핸들러 이벤트 호출
	if mysqlServerPort >= 0 {
		srv.tcpListener, err = mysql.NewListener( // ✅ TCP 서버 생성
			// ...
		)
		
		// ...
		
		// Start listening for tcp
		// ✅ TCP 서버 실행
		go srv.tcpListener.Accept()
	}
	
	// ...
}
```

해당 함수의 역할은 몇 가지가 더 있지만, 현재 관심있는 부분은 TCP 서버를 여는 부분입니다. 해당 관점으로 살펴보면 이 함수의 역할은 단순하게 요약할 수 있습니다.

- TCP 연결 핸들러를 생성하고
- `mysql.NewListener`로 TCP 서버 생성하고
- goroutine으로 `tcpListener.Accept`을 호출합니다.

서버가 어떤 형태로 생성되는지 확인하고 싶다면 `mysql.NewListener`메서드를 확인하면 됩니다. 이 함수 내부를 확인해보면 어떻게 TCP 서버를 생성하고 있는지 확인할 수 있습니다.

```go
// https://github.com/vitessio/vitess/blob/b0352bd57c9fba3c4c1fb94f1ed59d5c7e54bf6d/go/mysql/server.go#L252
// NewListener creates a new Listener.
func NewListener(
	protocol, address string,
	authServer AuthServer,
	handler Handler,
	connReadTimeout time.Duration,
	connWriteTimeout time.Duration,
	proxyProtocol bool,
	connBufferPooling bool,
	keepAlivePeriod time.Duration,
	flushDelay time.Duration,
) (*Listener, error) {
	// ✅ net.Listen을 이용하여 golang 기본 net 패키지로 TCP 서버를 호스팅
	listener, err := net.Listen(protocol, address)
	if err != nil {
		return nil, err
	}
	
	// ...

	return NewFromListener(listener, authServer, handler, connReadTimeout, connWriteTimeout, connBufferPooling, keepAlivePeriod, flushDelay)
}
```

여기서 `net.Listen`을 호출하여 go의 기본적인 net 패키지를 활용하여 tcp 서버를 만든다는 것을 알 수 있습니다. 이렇게 go의 기본 라이브러리를 활용하여 Listener를 반환합니다. 이렇게 Listener를 생성하고 소켓 생성 이벤트와 여러 설정을 부여합니다. 다음은 `Listener.Accept`을 확인해보겠습니다. `Listener.Accept`은 TCP 서버에 연결된 커넥션을 확인하고 각 커넥션마다 goroutine을 할당하는 역할을 수행합니다.

```go
// https://github.com/vitessio/vitess/blob/b0352bd57c9fba3c4c1fb94f1ed59d5c7e54bf6d/go/mysql/server.go#L328
// Accept runs an accept loop until the listener is closed.
func (l *Listener) Accept() {
	ctx := context.Background()

	for {
		// ✅ 커넥션 연결 블로킹
		conn, err := l.listener.Accept()
		if err != nil { // error가 nil인 경우 Close() 명시 호출로 간주
			// Close() was probably called.
			connRefuse.Add(1)
			return
		}

		acceptTime := time.Now()

		connectionID := l.connectionID
		l.connectionID++

		// ✅ 커넥션 개수 기록
		connCount.Add(1)
		connAccept.Add(1)

		go func() {
			// ...
			
			// ✅ 실제 로직 함수 handle 호출
			l.handle(conn, connectionID, acceptTime)
		}()
	}
}
```

해당 함수 내부에서 `net.Listener`의 `Accept`을 호출하여 블로킹으로 소켓 연결을 대기하고 있습니다. 커넥션 연결이 수락됐다면 연결 수락 시간을 기록한 후, 커넥션 아이디를 기록하고 소켓에 대한 핸들함수인 `l.handle`을 각 소켓마다 goroutine으로 실행합니다. 여기서 클라이언트의 연결이 N개이면, 실행되는 클라이언트 연결에 대한 goroutine도 N개라는 것을 알 수 있습니다. 또한 코드를 확인했을 때 mysql 서버에 쿼리를 처리하는 부분은 `l.handle`함수 내부에 존재한다는 것을 알 수있습니다.

다음 아티클은 `l.handle`함수 내부를 살펴보고 연결된 소켓이 어떻게 처리되는지 확인해보겠습니다.
