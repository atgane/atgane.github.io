application controller에서 application에 대한 이벤트를 informer가 수신하여 CUD에 대한 연산을 큐로 넘기고 `~~~QueueItem()`함수가 비동기로 처리하는 부분까지 확인했습니다. App이 새로 추가된 경우, informer의 `AddFunc`이 호출되고 생성된 object에 대한 key를 다른 고루틴으로 넘겨 처리합니다. 이 고루틴은 `processAppRefreshQueueItem` 함수를 실행합니다. 이번 아티클에서 `processAppRefreshQueueItem`이 함수 내부를 탐색해보겠습니다.

---

# processAppRefreshQueueItem 내부

`processAppRefreshQueueItem`은 CUD에 대한 연산이 수행된 app을 비동기로 처리하는 최상위 함수입니다. 이 함수가 수행하는 동작이 많지만 app 생성 과정에서 크게 요약해보면 다음과 같이 작업이 이루어집니다.

```go
// https://github.com/argoproj/argo-cd/blob/079754c63913522803f7dbe4bded6b6de37f7e34/controller/appcontroller.go#L1541
func (ctrl *ApplicationController) processAppRefreshQueueItem() (processNext bool) {
	
	// ...
	
	// ✅ queue에 삽입된 원소를 구독
	appKey, shutdown := ctrl.appRefreshQueue.Get()
	
	// ...
	
	// ✅ 1. app 상태가 refresh가 필요한 상태인지 검증
	needRefresh, refreshType, comparisonLevel := ctrl.needRefreshAppStatus(origApp, ctrl.statusRefreshTimeout, ctrl.statusHardRefreshTimeout)
	
	// ... 
	
	// ✅ 2. project 조회 및 spec 검증
	project, hasErrors := ctrl.refreshAppConditions(app)
	
	// ...
	
	// ✅ 3. app 상태 비교
	compareResult, err := ctrl.appStateManager.CompareAppState(app, project, revisions, sources,
		refreshType == appv1.RefreshTypeHard,
		comparisonLevel == CompareWithLatestForceResolve, localManifests, hasMultipleSources, false)
		
	// ...
	
	// ✅ 4. app에 대한 tree 생성 및 summary 작성
	tree, err := ctrl.setAppManagedResources(app, compareResult)
	
	// ...
	
	// ✅ 5. sync 가능 상태에 따라 autosync
	canSync, _ := project.Spec.SyncWindows.Matches(app).CanSync(false)
	if canSync {
		syncErrCond, opMS := ctrl.autoSync(app, compareResult.syncStatus, compareResult.resources, compareResult.revisionUpdated)
		// ...
	}
	
	// ...
	// app 상태 갱신 및 마무리
}
```

가장 먼저 queue에서 원소를 가져옵니다. queue에서 원소를 가져온 이후 단계는 크게 다섯 단계로 구분할 수 있습니다.

1. app 상태가 refresh가 필요한지 검증합니다. 강제 or 일반 refresh와 리소스를 어떻게 동기화할지 레벨을 결정합니다.
2. project 조회 및 app spec 검증합니다. app의 형식이  argocd에 포함된 클러스터인지 원격 저장소는 접근 가능한지 여부를 검증합니다. 
3. app 상태 비교: 가장 중요한 부분으로 원격 저장소의 상태와 k8s로 관리되는 app의 리소스를 비교하는 과정을 수행합니다.
4. app에 대한 tree를 생성하고 app summary를 작성합니다.
5. sync가능 여부에 따라 autosync를 활성화합니다.

`processAppRefreshQueueItem` 내부에서 가장 중요한 부분은 3번째 과정인 app 상태를 비교하는 부분입니다. `ctrl.appStateManager.CompareAppState` 함수로 argocd에서 실질적인 gitops가 구현됩니다. 그럼 `ctrl.appStateManager.CompareAppState`내부를 살펴보겠습니다.

```go
// https://github.com/argoproj/argo-cd/blob/079754c63913522803f7dbe4bded6b6de37f7e34/controller/state.go#L429
func (m *appStateManager) CompareAppState(app *v1alpha1.Application, project *v1alpha1.AppProject, revisions []string, sources []v1alpha1.ApplicationSource, noCache bool, noRevisionCache bool, localManifests []string, hasMultipleSources bool, rollback bool) (*comparisonResult, error) {

	// ...

	
	if len(localManifests) == 0 {

		// ...
		
		// ✅ 1. 원격 저장소에서 대상 object 조회
		targetObjs, manifestInfos, revisionUpdated, err = m.GetRepoObjs(app, sources, appLabelKey, revisions, noCache, noRevisionCache, verifySignature, project, rollback)
		
		// ...
	
	}

	// ...
	
	//  ✅ 2. 중복 오브젝트 제거
	targetObjs, dedupConditions, err := DeduplicateTargetObjects(app.Spec.Destination.Namespace, targetObjs, infoProvider)
	
	// ...

	// ✅ 3. 대상 클러스터에 배포된 object 조회
	liveObjByKey, err := m.liveStateCache.GetManagedLiveObjs(app, targetObjs)
	
	// ...
	
	// 4. ✅ 재조정 대상 오브젝트 조회
	reconciliation := sync.Reconcile(targetObjs, liveObjByKey, app.Spec.Destination.Namespace, infoProvider)
	
	// ...
	
	// 5. ✅ diff config 설정 및 diff 결과 가져옴
	diffResults, err := argodiff.StateDiffs(reconciliation.Live, reconciliation.Target, diffConfig)
	
	// ...
	
	// 6. ✅ 리소스 순회하며 managedResources, resourcesSummaries 필드 생성
	managedResources := make([]managedResource, len(reconciliation.Target))
	resourceSummaries := make([]v1alpha1.ResourceStatus, len(reconciliation.Target))
	for i, targetObj := range reconciliation.Target {
		
		// ...
	
		if liveObj != nil {
			resourceVersion = liveObj.GetResourceVersion()
		}
		managedResources[i] = managedResource{
			Name:            resState.Name,
			Namespace:       resState.Namespace,
			Group:           resState.Group,
			Kind:            resState.Kind,
			Version:         resState.Version,
			Live:            liveObj,
			Target:          targetObj,
			Diff:            diffResult,
			Hook:            resState.Hook,
			ResourceVersion: resourceVersion,
		}
		resourceSummaries[i] = resState
	}
	
	// ...

	compRes := comparisonResult{
		syncStatus:           &syncStatus,
		healthStatus:         healthStatus,
		resources:            resourceSummaries,
		managedResources:     managedResources,
		reconciliationResult: reconciliation,
		diffConfig:           diffConfig,
		diffResultList:       diffResults,
		hasPostDeleteHooks:   hasPostDeleteHooks,
		revisionUpdated:      revisionUpdated,
	}
	
	// ...
	
	return &compRes, nil
}
```

함수 동작을 요약하면 다음과 같습니다.

- 원격 저장소에 있는 오브젝트 목록을 가져옵니다.
- 현재 배포 상태를 조회합니다.
- 재조정할 대상 오브젝트를 조회합니다.
- diff 범위를 설정합니다.
- 리소스를 순회하면서 상태를 비교합니다.

이 동작은 테스트 코드를 통해 확인이 가능합니다. 

```go
// https://github.com/argoproj/argo-cd/blob/079754c63913522803f7dbe4bded6b6de37f7e34/controller/state_test.go#L481
// TestAppRevisions tests that revisions are properly propagated for a single source app
func TestAppRevisionsSingleSource(t *testing.T) {
	obj1 := NewPod()
	obj1.SetNamespace(test.FakeDestNamespace)
	data := fakeData{
		// ✅ 원격 저장소에 존재하는 object: 파드를 하나 배포해야 함
		manifestResponse: &apiclient.ManifestResponse{
			Manifests: []string{toJSON(t, obj1)},
			Namespace: test.FakeDestNamespace,
			Server:    test.FakeClusterURL,
			Revision:  "abc123",
		},
		// ✅ k8s 대상 클러스터에 배포된 리소스: map에 아무 원소 없음
		managedLiveObjs: make(map[kube.ResourceKey]*unstructured.Unstructured),
	}
	ctrl := newFakeController(&data, nil)

	app := newFakeApp()
	revisions := make([]string, 0)
	revisions = append(revisions, "")
	compRes, err := ctrl.appStateManager.CompareAppState(app, &defaultProj, revisions, app.Spec.GetSources(), false, false, nil, app.Spec.HasMultipleSources(), false)
	// 통과 조건
	require.NoError(t, err)
	assert.NotNil(t, compRes)
	assert.NotNil(t, compRes.syncStatus)
	assert.NotEmpty(t, compRes.syncStatus.Revision)
	assert.Empty(t, compRes.syncStatus.Revisions)
}
```

위의 테스트 로직을 추적해보면 다음과 같이 동작합니다.

1. `CompareAppState` 의 ✅ 1번에서 `targetObjs`에 pod가 하나 포함됩니다. 또한 Manifests.Manifests 배열에 manifest가 string 형식으로 저장됩니다.
2. 중복 오브젝트는 건너뜁니다.
3. ✅ 3번에서 `liveObjByKey`로 배포된 오브젝트를 가져오지만 아무것도 조회되지 않습니다.
4. ✅ 4번에서 재조정 대상 오브젝트를 조회합니다. 이때 `reconciliation`에는 Live와 Target을 배열로 소유하는데, Live는 현재 배포된 상태, Target은 배포할 상태를 의미합니다. Live는 nil을 가지고 Target에는 파드를 가집니다.
5. ✅ 5번에서 diff결과를 만듭니다. `DiffResults`에는 `reconciliation`처럼 live와 target에 대한 정보를 string manifest로 소유합니다.

좀 더 복잡한 테스트를 확인하고 싶으면 `TestCompareAppStateDuplicatedNamespacedResources`가 좋은 예시가 되겠습니다.

이렇게 k8s에 app 오브젝트가 생성되어 informer로 이벤트를 받아 queue로 처리하고 CompareAppState함수로 오브젝트 상태를 파악하는 과정을 거쳤습니다. CompareAppState함수 내부에서 원격 저장소 상태와 클러스터 상태를 비교하여 원하는 상태를 확인하는 과정을 살펴봤습니다. 그러나 아직 살펴보지 않은 부분이 두 부분이 있습니다. 첫 번째는 repository server와 상호작용이고, 하나는 k8s 리소스에 대한 생성입니다. 다음에는 repository server와 상호작용하는 부분을 찾아보겠습니다.
